# 一人称視点ライン作業分析システム 設計ドキュメント（Ollo Factory ベンチマーク）

- **作成日**: 2026-06-09
- **ステータス**: 設計承認済み / 実装計画待ち
- **対象**: 工場ラインにおける一人称視点（ウェアラブル）映像のゼロショットアクションセグメンテーション基盤の MVP

---

## 1. 背景と狙い

### 1.1 ベンチマーク対象：Ollo Factory（株式会社Ollo）

東京大学 松尾研究室発のAIスタートアップによる製造業特化の作業分析AI。世界30カ国以上・5,000現場以上で導入実績があり、直近でスズキが正式導入、JR東日本グループ（車両製造）で実証実験中。3モジュール構成：

| モジュール | カイゼン領域 | 主な機能 |
|---|---|---|
| **Tools** | 標準化 | 手順書作成・ムダ作業分類・改善案提案 |
| **Training** | 習熟・定着 | ベテランと新人の比較・山積み表・ばらつき分析 |
| **Alert** | ミス防止 | リアルタイム異常検知・スマホ通知・原因映像記録 |

コア技術：ワンショット学習（1サイクル撮影で開始）、ゼロショットアクションセグメンテーション（テキストラベルから作業分類）、ヒント画像（□で囲うと物体を自動認識）、生成AIによる作業解説・改善提案。OpenAI / Gemini 等の汎用大規模モデルを活用していることが公表されている。

### 1.2 本プロジェクトの位置づけ

本システムは Ollo Factory に対して **(A) 競合・代替製品** であると同時に **(C) 研究・ベンチマーク用途** を兼ねる。すなわち：

- Ollo Factory 相当の機能を自前で持つシステムを構築する
- 一人称視点映像解析の最新手法（学会報告で高精度なもの）を積極的に取り込み、Olloや既存手法と精度比較できる評価基盤を持つ
- 注目学会：WACV / MVA / ETFA（実応用・FA特化の国際学会）、SSII / MIRU（国内の製造実用事例）、CVPR（基礎技術）

### 1.3 技術スタンスの前提

- **アプリケーション層を作る**：Gemini 等の既存大規模 VLM を推論基盤として使い、その上にパイプライン・分析ロジック・評価基盤を構築する。**モデル自体は自前で訓練しない**（Ollo と同じ土俵に揃える）
- 調査時点の知見：マルチモーダルなアクションセグメンテーションでは **Gemini の Pro モデルが SOTA 水準**。一人称視点ベンチマーク EgoInstruct（arXiv:2509.22019）で Gemini 2.5 Pro がステップセグメンテーション Acc 94〜95.6%、GPT-4o（80.7%）を大きく上回ると報告
- MIRU2025 チュートリアル「一人称視点映像解析の最先端」でも「現状は既存の汎用 MLLM を使うのがベスト」とされ、本方針を裏付ける

## 2. スコープ

### 2.1 MVP の目標

> **20分超クラスの長尺・非反復（single long procedure）の一人称視点ライン作業映像に対し、テキストラベル群を与えてゼロショットでアクションセグメンテーションを行う。**

- 80分クラスの映像は**別システムで〜20分に分割される前提**であり、本システムの責任範囲外
- 対象映像は**非反復**（段取り替え・多品種少量等で同一サイクルが繰り返されない）。したがってサイクル自己相似（recurrence）には依存せず、純粋な逐次セグメンテーション＋階層チャンキングで設計する
- 起点は Ollo の **Tools（標準化）** 系。Training（習熟・定着）/ Alert（ミス防止）は後続フェーズ

### 2.2 入出力

| 項目 | 内容 |
|---|---|
| 入力（映像） | 20分超クラスの長尺・非反復ライン作業映像（一人称視点、ウェアラブル撮影） |
| 入力（ラベル） | 利用者が事前定義する要素作業の語彙（テキストラベル群）。Ollo の「事前ラベル」相当 |
| 入力（ヒント画像・将来） | 物体を□で囲ったヒント画像（次フェーズ。設計上の拡張枠のみ確保） |
| 出力 | 時系列セグメント列（開始/終了時刻 ＋ 作業ラベル ＋ 信頼度）。JSON 形式 |
| 出力（派生） | タイムライン可視化、手順書ドラフト（Markdown/HTML） |

### 2.3 スコープ外（MVPに含めないが、後続フェーズで実装予定）

| 除外項目 | 扱い |
|---|---|
| 80分一括処理 | 別システムで分割する前提。自システムは20分超まで対応 |
| リアルタイム推論 | 将来フェーズ。Stage 1 をストリーミング化できる設計を「想定」として残す |
| Alert（異常検知） | 後続フェーズ |
| Training系（山積み表・ばらつき・動画間差分） | 後続フェーズ。サイクル反復前提機能のため非反復MVPの後 |
| ヒント画像による物体追跡 | 後続フェーズ。Stage 2 に拡張インターフェースの「枠」のみ用意 |
| 生成AIによる改善提案 | 後続フェーズ。セグメンテーション精度が担保されてから乗せる |
| UIフロントエンド | 後続フェーズ。MVP は CLI/スクリプトで動作すれば十分 |

## 3. アーキテクチャ

### 3.1 全体パイプライン（階層型 Coarse-to-Fine）

20分・非反復向けに、**境界検出と意味付けを分離**した階層構成を採る。長尺をVLMに丸投げするとタイムスタンプのハルシネーション／区間取りこぼしが起きる（NOAH等で報告）ため、生の長尺全体はVLMに見せない。

```
一人称視点映像（〜20分超・非反復）
  ↓
[Stage 0] 取り込み・前処理
   ・フレーム抽出（1fps基準、可変）
   ・顔ブラー等プライバシー処理（EgoBlur / InsightFace）
  ↓
[Stage 1] 粗いプレセグメンテーション ← VLM不使用・全編を安価にスキャン
   ・フレーム埋め込み（SigLIP/CLIP）の時系列に変化点検出（ruptures）
   ・非反復のため境界候補を逐次的に列挙（自己相似は使わない）
  ↓
[Stage 2] ゼロショット作業ラベリング
   ・各区間 × テキストラベル の類似度（CLIP/VLM）で作業分類
   ・境界候補まわりの窓だけVLMで局所精緻化（局所チャンキング）
  ↓
[Stage 3] 出力整形
   ・セグメント列（JSON）→ タイムライン / 手順書ドラフト
```

### 3.2 設計上の中核原則

1. **境界検出（Stage 1）と意味付け（Stage 2）を分離** — 長尺ハルシネーションを回避し、各段を独立にベンチマークできる
2. **Stage 2 を差し替え可能なインターフェースにする** — 複数のセグメンテーション手法を同一の評価軸で比較できる（下記 Track A/B、将来は学術手法 PGNet 等）
3. **コストを尺にスケールさせない** — 高価なVLM呼び出しは境界候補まわりの窓に限定し、全フレーム投入を避ける
4. **将来のリアルタイム化を想定** — Stage 1 をストリーミング処理に置換できる構造にしておく（MVPでは実装しない）

### 3.3 ゼロショットセグメンテーションの2トラック同居

2つの実装を**同居**させ、評価基盤上で競わせる（ベンチマークの中核価値）。

- **Track A（ベースライン）**：Gemini Pro（最新 3.x 系）単一投入型
  - 20分（約31万トークン、概算）を窓分割し、各窓にラベル列＋時刻を出力させて stitch（結合）する
  - EgoInstruct で SOTA 水準（Acc 94%超）が確認されている強力なベースライン
- **Track B（パイプライン）**：Stage 1 境界候補 × Stage 2 ゼロショット分類型
  - コスト効率と長尺耐性で Track A を上回ることを狙う
  - Stage 2 差し替えインターフェースの最初の実装

両トラックは**同一の入力（映像＋ラベル語彙）と同一の出力形式（セグメント列JSON）**を持ち、同じ評価スクリプトで測定する。

## 4. 評価・ベンチマーク基盤

### 4.1 評価方針

自社ライン映像を手動アノテーションして正解とし、システム出力をそれと照合する。同じ映像に Ollo を流し、出力を並べて定量・定性比較する。

### 4.2 評価フロー

1. **自社映像の収集** — 対象ラインで一人称視点映像を撮影（20分超クラス）。まず 1〜3 本から開始
2. **アノテーション** — CVAT（OSSの動画アノテーションツール、セルフホスト可）で要素作業レベルのラベルを時系列付与
3. **評価スクリプト** — 標準的な時系列アクションセグメンテーション指標を自動算出
   - セグメンタル F1@{10, 25, 50}
   - セグメンタル Edit 距離
   - フレーム単位 Accuracy
4. **3系統の横並び比較** — Track A / Track B / Ollo を同一指標で並べる
   - Ollo の出力セグメント列をエクスポートし、正解形式に変換して同スクリプトで計測

### 4.3 アノテーション設計

- 粒度は**要素作業レベル**で統一（Ollo の "要素作業" 相当）。細かすぎず粗すぎず
- アノテーションのラベル語彙を、システムへのテキストラベル入力にもそのまま使う（ゼロショット評価として正しい条件）
- **拡張口**：将来 Assembly101（製造組立・非反復・1M+セグメント・正解ラベル付き）や Ego4D Goal-Step を評価対象に追加すると、学会論文の数値と直接対比できる。評価スクリプトは公開データセット形式も読めるよう汎用化しておく

## 5. 技術スタック

| 層 | 採用候補 | 理由 |
|---|---|---|
| フレーム埋め込み | SigLIP / CLIP（ViT-L/14） | ゼロショット類似度が強い。Stage 1・2 両方で使える |
| 変化点検出 | `ruptures`（Python OSS） | 時系列変化点検出ライブラリ。実績あり |
| VLM API | Gemini Pro（最新の 3.x 系を採用。Google AI SDK） | マルチモーダルなアクションセグメンテーションで SOTA 水準とされる。動画ネイティブ入力対応。※公開ベンチマークの実測値は前世代 2.5 Pro で報告（EgoInstruct Acc 94〜95.6%）であり、本システムは入手可能な最新 Pro モデルを採用しつつ評価で性能を実測する |
| 動画処理 | OpenCV + ffmpeg | フレーム抽出・リサイズ・顔ブラー |
| プライバシー処理 | EgoBlur / InsightFace | 一人称視点向け顔匿名化。MIRU2025 チュートリアルでも言及 |
| アノテーション | CVAT（セルフホスト） | OSS。工場でオンプレ運用しやすい |
| バックエンド | Python + FastAPI | バッチ処理パイプライン ＋ 将来のAPI化 |
| 出力フォーマット | JSON（セグメント列）＋ Markdown/HTML（手順書ドラフト） | Olloの手順書出力と並べやすい |

## 6. データフローとインターフェース

### 6.1 セグメント列の共通スキーマ（Track A / B / Ollo / 正解 共通）

全系統がこの形式に揃うことで、評価スクリプトが系統非依存になる。

```json
{
  "video_id": "line01_20min_001",
  "fps_sampled": 1.0,
  "label_vocabulary": ["部品取り出し", "ネジ締め", "検査", "..."],
  "segments": [
    { "start_sec": 0.0,  "end_sec": 12.4, "label": "部品取り出し", "confidence": 0.87 },
    { "start_sec": 12.4, "end_sec": 30.1, "label": "ネジ締め",     "confidence": 0.79 }
  ],
  "source": "track_b"
}
```

### 6.2 コンポーネント境界

各コンポーネントは単一責務とし、独立に理解・テストできる単位に分割する。

| コンポーネント | 責務 | 依存 |
|---|---|---|
| `ingest` | 映像→フレーム列＋メタデータ、プライバシー処理 | OpenCV/ffmpeg, EgoBlur |
| `embed` | フレーム列→埋め込み系列 | SigLIP/CLIP |
| `presegment` | 埋め込み系列→境界候補（Stage 1） | ruptures |
| `label_zeroshot` | 区間＋ラベル語彙→ラベル付きセグメント（Stage 2 / Track B） | CLIP, Gemini |
| `label_vlm_single` | 映像窓＋ラベル語彙→ラベル付きセグメント（Track A） | Gemini |
| `evaluate` | 予測セグメント列＋正解→指標 | — |
| `report` | セグメント列→タイムライン/手順書ドラフト | — |

`label_zeroshot` と `label_vlm_single` は共通インターフェース（入力＝映像＋ラベル語彙、出力＝セグメント列）を実装し、差し替え可能にする。将来の学術手法やヒント画像追跡もこのインターフェースに乗せる。

## 7. リスクと留意点

| リスク | 対応 |
|---|---|
| Stage 1 の境界検出精度が低いと Track B 全体が劣化 | 境界候補は recall 重視（過分割気味）で出し、Stage 2/VLM で統合・精緻化 |
| 20分のGemini単一投入はコストが高い | Track A はベースライン位置づけ。常用は Track B。窓分割でトークン上限を管理 |
| アノテーション工数 | 要素作業粒度を固定し、まず1〜3本に限定。語彙を先に確定してブレを抑える |
| Ollo出力の正解形式への変換 | 変換アダプタを独立コンポーネント化し、Ollo のエクスポート仕様変更に追従しやすくする |
| ラベル語彙の設計がゼロショット精度を左右 | 語彙設計を評価の独立変数として記録し、再現性を担保 |

## 8. 段階計画（参考）

- **フェーズ1（本MVP）**：Stage 0–3、Track A/B、評価基盤、自社映像1〜3本での Ollo 比較

- **フェーズ3**：Training系（山積み表・ばらつき・動画間差分）
- **フェーズ4**：リアルタイム化（Stage 1 ストリーミング）、Alert 相当、UI

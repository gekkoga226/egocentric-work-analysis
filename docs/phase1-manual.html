Phase 1 概要

Phase 1 は CLIで動作するバックエンドパイプライン です。工場ラインの一人称視点動画（20分前後）を入力し、どの時間帯にどの作業をしていたかを自動でセグメンテーションします。

🎯
目的：Ollo Factory Tools と同等機能を自前実装し、精度をベンチマーク比較できる基盤を作る。Phase 2（Web UI）は Phase 1 の動作確認後に着手。

Phase 1 で実現すること

機能状態説明
フレーム抽出✅ 完了動画から指定FPSでフレームをサンプリング。顔ぼかし対応。
Track B：CLIPパイプライン✅ 完了CLIP埋め込み → 変化点検出 → ゼロショットラベリング
Track A：Gemini単一パス✅ 完了5分ウィンドウ分割でGemini 2.5 Proにフレーム送信
TAS評価指標✅ 完了F1@{10,25,50}・Edit Distance・フレーム精度
マルチシステム比較✅ 完了Track A / B / Ollo を同一指標で横並び比較
CLIスクリプト✅ 完了run_pipeline.py / run_evaluate.py
Web UIPhase 2FastAPI + htmx + Alpine.js（設計済み、未実装）

Track B を実行する（CLIP パイプライン）

✅
まずここから始めてください。Gemini API キー不要。初回のみ CLIP モデルをダウンロード（約600MB）します。

基本コマンド

PowerShell 
cd C:\Users\mizuk\work\ollo-benchmark

python -m scripts.run_pipeline `
  C:\path\to\your\video.mp4 `
  "部品取り出し,ネジ締め,検査,梱包" `
  --track b

何が検証できるか

🔬 パイプライン全体の疎通

  • フレーム抽出が動作するか
  • CLIPモデルが正常にロードされるか
  • 変化点検出が境界を返すか
  • ラベルが正しく割り当てられるか
  • JSONファイルが results/ に保存されるか

🎯 ゼロショット性能の確認

  • 定義したラベル語彙で正しく分類できるか
  • 作業の切り替わりタイミングを検出できるか
  • 意味的に異なる作業を区別できるか
  • CLIPの特徴空間がファクトリー映像に適用できるか

⚙️ パラメータの挙動確認

  • --fps 変更で分解能と速度のトレードオフ
  • --penalty 変更で分割粒度の調整
  • 短い動画で動作 → 長い動画（20分）でスケールするか

📊 出力形式の確認

  • Markdownタイムラインが可読か
  • JSONスキーマが正しいか
  • タイムスタンプが動画長と整合するか

詳細オプションの使い方

PowerShell 
# FPSを上げて精度向上（処理時間も増加）
python -m scripts.run_pipeline video.mp4 "作業A,作業B,作業C" `
  --track b --fps 2.0

# 変化点の感度を上げる（penaltyを小さく → セグメント数増加）
python -m scripts.run_pipeline video.mp4 "作業A,作業B,作業C" `
  --track b --penalty 5.0

# 顔ぼかし有効（プライバシー保護）
python -m scripts.run_pipeline video.mp4 "作業A,作業B,作業C" `
  --track b --blur-faces

# 出力先を指定
python -m scripts.run_pipeline video.mp4 "作業A,作業B,作業C" `
  --track b --output-dir C:\results\experiment1

何が検証できるか

🤖 VLM推論の品質

  • Gemini 2.5 Pro が工場映像を正しく理解できるか
  • 5分ウィンドウの分割・マージが正しく機能するか
  • プロンプトに対して正しいJSON形式で返答するか
  • confidenceスコアが意味のある値を返すか

⚡ エラー耐性

  • API失敗時に警告ログを出してスキップするか
  • レスポンスにJSON配列がない場合を正しく処理するか
  • 長尺動画（20分 = 4ウィンドウ）を完走するか

両方を同時実行

  C:\path\to\your\video.mp4 `
  "部品取り出し,ネジ締め,検査,梱包" `

何が検証できるか

  • 同一動画に対して Track A と Track B の結果が results/ に並んで生成されるか
  • 両者のセグメント境界・ラベルの傾向の違いを目視確認できる
  • 後続の評価コマンドで定量比較するための準備が整う

生成される出力ファイル

results/ ├── video_track_b.json # Track B の結果 └── video_track_a.json # Track A の結果

評価指標で精度を測る

正解アノテーション（手動ラベリング）と比較して、定量スコアを算出します。

評価指標の説明

F1@10 F1@25 F1@50 Edit Distance Frame Accuracy
指標計算方法何を測るか理想値
F1@k
k=10,25,50		 予測セグメントと正解セグメントのIoUがk%以上の場合にTPとカウント。Precision・Recallを算出してF1を計算。 セグメント境界の正確さ。k=50は「半分以上重なっていれば正解」、k=10は「少し重なっていれば正解」。 1.0（=100%）
Edit ラベル列のLevenshtein距離を最大長で正規化して0〜100に変換。 セグメント順序・ラベル系列の正しさ。境界がずれていても順序が合っていれば高スコア。 100.0
Acc 1秒ごとにフレームラベルを比較して一致率を計算。 フレーム単位の正確さ。長いセグメントが多い動画で高くなりやすい。 1.0（=100%）

評価コマンド

PowerShell 
# Track B のみ評価
python -m scripts.run_evaluate `
  annotations\video.json `
  results\video_track_b.json

# Track A / B を横並び比較
python -m scripts.run_evaluate `
  annotations\video.json `
  results\video_track_b.json `
  results\video_track_a.json

出力例

| system | f1@10 | f1@25 | f1@50 | edit | acc | |---------|-------|-------|-------|-------|-------| | track_b | 0.85 | 0.80 | 0.70 | 75.0 | 0.82 | | track_a | 0.90 | 0.87 | 0.78 | 82.0 | 0.88 |

何が検証できるか

📊 Track A vs Track B の優劣

  • 同一動画で Track A / B のスコアを比較
  • コスト・速度・精度のトレードオフを定量化
  • 作業の種類によって得意不得意が変わるか確認

🔧 パラメータ調整の効果

  • --penalty を変えて F1 がどう変わるか
  • --fps を上げると Acc が向上するか
  • 最適パラメータを探索できる

🏭 Ollo との比較基盤

  • Ollo の出力をJSONに変換して同指標で評価可能
  • 学術論文と同一指標（TAS標準）で比較できる
  • Assembly101 / Ego4D 等のベンチマークへの拡張が可能

🔬 評価指標の妥当性確認

  • F1@10 > F1@50 が必ず成り立つか
  • 完全一致の予測でスコアが全て最大値になるか
  • 完全不一致でスコアが最小になるか

オプション一覧

オプションデフォルト説明

何が検証できるか — まとめ

コマンド検証できること前提
pytest tests/ -v --ignore=... 全モジュールの単体テスト（37件）。スキーマ・評価指標・パイプライン各ステージの正確性をコード変更前後に確認。 CLIPモデル不要
run_pipeline --track b Track B パイプライン全体の疎通。CLIP が工場映像を分類できるかの実映像検証。penalty/fps チューニング。 CLIP DL（初回）
run_pipeline --track a Gemini 2.5 Pro の工場映像理解能力。5分ウィンドウ分割の動作。JSON出力の安定性。 GEMINI_API_KEY
run_pipeline --track both Track A / B を同一動画で比較実行。両者の結果ファイルを並べて目視・定量比較する準備。 両方
run_evaluate gt.json pred_b.json pred_a.json TAS 標準指標（F1@10/25/50・Edit・Acc）での定量評価。Track A vs Track B の精度差。Ollo との比較基盤。 正解アノテーション

推奨する検証ステップ

  1. テスト実行 — pytest で 37件パスを確認
  2. 短い動画で Track B 動作確認 — 2〜3分の動画で疎通テスト
  3. 20分動画で Track B 実行 — 長尺動画のスケールを確認
  4. 手動でアノテーション作成 — 目視で正解タイムスタンプを記録
  5. 評価コマンドで定量スコア算出 — F1・Edit・Acc を確認
  6. パラメータを変えて再実行 — penalty / fps を調整してスコアの変化を確認
  7. Track A と比較 — APIキー取得後、同動画で Track A を実行して定量比較
🚀
Phase 2 へ進む条件： Track B で実動画に対して有意なセグメンテーション結果が得られ、評価指標が安定したら Phase 2（Web UI）の実装に着手します。
